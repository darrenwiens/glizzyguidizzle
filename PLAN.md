## 1. Tech Stack & Architecture

### Core Technologies
| Layer | Choice | Rationale |
|-------|--------|-----------|
| Language | Vanilla JavaScript (ES2022+) | No framework overhead, native ES modules |
| Build Tool | Vite (optional) or none | Zero-config dev server; can run without build step |
| Map + Street View | Google Maps JavaScript API v3 | Single SDK for map tiles, markers, and Street View |
| Marker Clustering | @googlemaps/markerclusterer | Official Google library, AdvancedMarkerElement support |
| Annotation | HTML5 Canvas API | Built-in, zero dependencies |
| Database | Supabase (PostgreSQL) | Real-time subscriptions, row-level security, JS client |
| Styling | Plain CSS with custom properties | BEM-lite naming, no preprocessor needed |
| State Management | Central state object + EventBus | Lightweight pub/sub, no framework magic |

### State Management Pattern

**Central state object with subscriber notifications:**
```javascript
// AppState.js
class AppState {
  constructor() {
    this.state = {
      locations: [],
      selectedLocationId: null,
      ui: { isMapView: true, isLoading: false, errorMessage: null },
      filters: { status: 'all', searchTerm: '' },
    };
    this.subscribers = new Map();
  }

  setState(updates) {
    this.state = { ...this.state, ...updates };
    this.notifySubscribers(Object.keys(updates));
  }

  subscribe(path, callback) { /* ... */ }
}
```

**EventBus for cross-module communication:**
```javascript
// EventBus.js ŌĆö events: 'marker-clicked', 'inspection-saved', 'street-view-loaded', etc.
eventBus.emit('marker-clicked', { locationId: 123 });
eventBus.on('inspection-saved', (data) => { /* update marker */ });
```

## 2. Data Model & Persistence

### Supabase Schema: `cantire_loc`
| Column | Type | Description |
|--------|------|-------------|
| `id` | int (PK) | Primary key |
| `latitude` | float | Store latitude |
| `longitude` | float | Store longitude |
| `name` | text | Store name |
| `address` | text | Full address |
| `category` | text | POI category |
| `stand_bool` | boolean (nullable) | `null` = unvisited, `true` = found, `false` = not found |
| `stand_bbox` | float[] | Bounding box coordinates `[x1, y1, x2, y2]` |
| `stand_view` | text | Street View panorama URL |

### JavaScript Data Model
```javascript
class Location {
  constructor(row) {
    this.id = row.id;
    this.latitude = row.latitude;
    this.longitude = row.longitude;
    this.name = row.name;
    this.address = row.address;
    this.category = row.category;
    this.standBool = row.stand_bool;   // null | true | false
    this.standBbox = row.stand_bbox;   // [x1, y1, x2, y2] or null
    this.standView = row.stand_view;   // panorama URL or null
  }

  getStatus() {
    if (this.standBool === null) return 'unvisited';
    return this.standBool ? 'found' : 'not-found';
  }

  getMarkerColor() {
    const colors = { found: '#10b981', 'not-found': '#ef4444', unvisited: '#9ca3af' };
    return colors[this.getStatus()];
  }
}
```

### Supabase Integration
```javascript
// Initialize client
import { createClient } from '@supabase/supabase-js';
const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY);

// Load all locations
const { data } = await supabase.from('cantire_loc').select('*');

// Update inspection result
await supabase.from('cantire_loc')
  .update({ stand_bool: true, stand_bbox: [x1,y1,x2,y2], stand_view: panoramaUrl })
  .eq('id', locationId);
```

### Caching Strategy
- **In-memory cache**: All ~500 locations loaded once on startup, cached in `AppState`
- **Local updates**: When user saves an inspection, update memory cache immediately + write to Supabase
- **No full reload needed**: Per-location updates only

## 7. Performance Strategy

### Map Marker Rendering (~500 Markers)
- **Google Maps MarkerClusterer** with `SuperClusterAlgorithm`: `radius: 80`, `maxZoom: 15`
- **AdvancedMarkerElement** (not legacy Marker): GPU-accelerated CSS transforms, 40-60% faster pan/zoom
- **Custom SVG markers inlined as data URIs**: ~200 bytes each, zero HTTP overhead
- **Lazy-render info windows**: Only create on click, not on mount
- **Performance**: ~500 markers ŌåÆ ~50-60 cluster groups, ~8-12ms render time

### Street View Loading
- **Google Maps API loaded once via `<script async defer>`** ŌĆö includes Street View service
- **Panorama created on demand**: New `StreetViewPanorama` only when user clicks a marker
- **Destroy on close**: `setPano(null)` to free memory when returning to map
- **Metadata preloading**: `StreetViewService.getPanorama()` on hover to pre-resolve panorama data

### Script Loading Order
```html
<!-- Google Maps API: async, loads independently -->
<script src="https://maps.googleapis.com/maps/api/js?key=KEY&libraries=marker,streetView"
        async defer></script>

<!-- Supabase client: async via CDN -->
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"
        async defer></script>

<!-- MarkerClusterer: async -->
<script src="https://unpkg.com/@googlemaps/markerclusterer@2.5.1/dist/index.min.js"
        async defer></script>

<!-- App: ES module, deferred -->
<script type="module" defer src="src/main.js"></script>
```

### Caching Strategy
| Layer | Content | TTL |
|-------|---------|-----|
| In-memory | All ~500 locations from Supabase | 30 min (then refresh) |
| Memory (per-update) | Individual location updates | Immediate (write-through to Supabase) |
| Browser HTTP cache | Google Map tiles | Auto-managed by browser |
| SessionStorage | Street View panorama metadata | Session duration |

### Core Web Vitals Targets
| Metric | Target | Strategy |
|--------|--------|----------|
| LCP | < 2.5s | Inline critical CSS, async script loading, cluster markers |
| INP | < 200ms | AdvancedMarkerElement (CSS transforms), lazy info windows |
| CLS | < 0.1 | Pre-allocate map container height, fixed-position inspection overlay |
| FCP | < 1.5s | Inline above-the-fold CSS, defer non-critical stylesheets |

## 9. Open Questions & Risks

### Google Maps API Costs
- **Free tier ($200/month credit)** should cover ~500 locations comfortably, but monitor usage via Google Cloud Console
- Heavy repeat usage (many users, many Street View loads) could approach limits
- Consider rate-limiting or caching panorama metadata to reduce API calls

### Street View Coverage
- Some rural Canadian Tire locations may lack Street View imagery
- Need fallback UX: satellite image + "Unable to Verify" status option
- Consider adding an "Unable to Verify" value as a fourth state (requires schema discussion)

### Bounding Box Coordinate System
- Canvas bounding box coordinates are **viewport-relative pixels**, not geo-coordinates
- When saving to `stand_bbox`, also save the panorama heading, pitch, and zoom in `stand_view` URL
- Without this context, the bounding box position is meaningless on a different view angle

### Supabase Row-Level Security
- The app writes directly to `cantire_loc` ŌĆö need Supabase RLS policies to prevent unauthorized writes
- Consider anonymous access for reads, authenticated access for writes (Supabase anon key vs auth)

### Data Quality
- Locations with missing/invalid coordinates should be skipped gracefully
- Log data quality issues for admin review

### Mobile Drawing Precision
- Touch-based bounding box drawing is inherently imprecise on small screens
- Tap-two-corners approach mitigates this, but user testing recommended
- Making annotation optional (status-only flow) covers 80% of use cases

### Offline Behavior
- App requires network for Google Maps tiles and Supabase reads
- Inspection results could be queued locally and synced when online (future enhancement)
- Service worker could cache map tiles for basic offline map viewing
